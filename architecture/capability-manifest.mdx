# Capability Manifest System

**Status: Specified**

Every entity in the HxTP ecosystem — physical device, digital agent, tool plugin — declares its capability set before it can receive commands. The capability registry is the authoritative record of what each entity is permitted to do.

## Core Concept

A **capability** is a named set of actions with defined parameter schemas and safety classifications. Capabilities are declared at provisioning time and may not be mutated at runtime without a versioned re-registration event.

The capability registry is not dynamic. It is not inferred from what a device can technically do. It is explicitly declared and becomes part of the device's immutable identity.

## Tool Manifest Schema

Tools (integrations that extend HX47's capabilities) declare themselves via a manifest:

```json
{
  "name": "@hxtp/tool-smart-plug",
  "version": "1.0.0",
  "sdk_version": ">=1.0.0",
  "capabilities": ["switch"],
  "actions": {
    "turn_on": {
      "description": "Turn on the smart plug",
      "params": {
        "type": "object",
        "properties": {
          "duration_ms": {
            "type": "integer",
            "description": "Optional: turn off after this duration"
          }
        }
      },
      "safety_class": "normal"
    },
    "turn_off": {
      "description": "Turn off the smart plug",
      "params": {
        "type": "object",
        "properties": {}
      },
      "safety_class": "normal"
    }
  },
  "supports_dry_run": false
}
```

## Safety Classifications

Every action belongs to one of four safety classes. This determines the validation gates and signatures required:

| Class | Description | Execution Path | Requires Safety Signature |
|---|---|---|---|
| **normal** | Non-mutating or low-risk operations (reads, state queries, light control). | Schema validation → policy pass → sign → dispatch | No |
| **sensitive** | Write operations, process spawning, form submission, external data transmission. | + Confirmation gate required | No |
| **critical** | Delete, deploy, credential access, security actuators (locks, alarms). | + Safety countersignature required. Dry-run if supported. | **Yes** |
| **restricted** | Raw shell, credential mutation, network configuration. | Human-authorized only. LLM cannot initiate. | **Yes** |

**Status of `restricted` class:** **Specified** — Reserved for future digital execution domain. Currently not applicable to physical Helix Nodes.

## Capability Declaration Invariants

1. **Immutability Between Versions:** Capabilities are immutable between versioned re-registration events.

2. **No Dynamic Mutation:** Dynamic capability mutation at runtime is prohibited. A device cannot add or remove capabilities without a versioned re-registration event that is logged and audited.

3. **Action Enum Construction:** The action enum presented to the LLM is not static. It is constructed at request time by the deterministic engine:
   - Identify device IDs relevant to the current request context
   - Query the capability registry for those device IDs
   - Extract the valid action set for each relevant capability
   - Build a per-request enum containing only those actions
   - Inject the enum into the tool schema passed to the LLM

4. **Scope Awareness:** The LLM never sees actions for devices outside the current request scope. The LLM never constructs action names. It selects from the provided enum.

5. **Registry Enforcement:** An action not in the capability registry for a device cannot be executed against that device, regardless of what the LLM proposes.

6. **Plugin Boundary:** Plugin tools cannot modify the capability registry schema. They declare into it.

## Capability Registry Model

The capability registry is a persistent data structure (Postgres table) that tracks:

```
Device/Agent Registry Entry:
├── device_id (UUID)
├── device_type (helix_node | digital_agent | external_api)
├── registered_capabilities
│   ├── capability_name (string)
│   ├── version (string)
│   ├── actions
│   │   ├── action_name
│   │   ├── schema (JSON Schema)
│   │   └── safety_class
│   └── registration_timestamp
├── created_at
├── updated_at
├── update_event_id (audit trail)
└── tenant_id
```

## Capability Update Events

When a capability is re-registered (version bumped), the system:

1. Records a new `capability_registration` event in the audit log
2. Creates a new capability registry entry with updated version
3. Increments the device's capability version counter
4. Notifies dependent systems (Safety Service, Edge Service) of the update
5. Previous version remains in audit history but is not active

This ensures:
- **Auditability:** Every capability change is logged
- **Rollback:** Previous versions are recoverable
- **Validation:** Intents issued against a specific capability version can be verified

## Device Provisioning and Capability Declaration

At provisioning time (during QR-code claim flow):

1. Device advertises its firmware version, device type, and declared capabilities
2. Edge Service validates against a known good manifest for that firmware version
3. Capability registry is populated with the declared set

## Firmware SDK Capability Declaration (hxtp-micro)

Embedded devices declare capabilities in firmware code using the SDK API:

```cpp
#include <hxtp.h>

HxDevice device("living_room_light");

device.capability("switch")
      .on("turn_on", [](){ 
          relay.high(); 
          return HxResult::SUCCESS; 
      })
      .on("turn_off", [](){ 
          relay.low();  
          return HxResult::SUCCESS; 
      })
      .safety_class("normal");

device.capability("dimmer")
      .on("set_brightness", [](const HxParams& p){ 
          int brightness = p.get<int>("level");
          pwm.set(brightness);
          return HxResult::SUCCESS;
      })
      .safety_class("normal");
```

The SDK handles:
- Encoding capability declarations in the provisioning advertisement
- Validating incoming commands against declared capabilities
- Rejecting unknown actions
- Reporting action availability to the Edge Service

## Cloud/Integrator Capability Declaration (hxtp-js)

Tools and integrations declare capabilities at registration time:

```typescript
import { HxCapability, HxAction } from 'hxtp-js';

const switchCapability: HxCapability = {
  name: 'switch',
  actions: {
    turn_on: {
      description: 'Turn on',
      params: {},
      safety_class: 'normal'
    },
    turn_off: {
      description: 'Turn off',
      params: {},
      safety_class: 'normal'
    }
  }
};

const tool = new HxTool('smart_plug')
  .declare(switchCapability)
  .register();
```

## Per-Request Action Enum Construction

When HX47 processes a user intent, the deterministic engine:

1. Extracts device IDs from the user's intent context
2. Queries the capability registry for those devices
3. Filters to actions with safety_class ≤ the user's authorization level
4. Constructs a per-request tool schema with only relevant actions
5. Injects this schema into the LLM's context

**Example:**

```json
{
  "user_intent": "turn on the lights",
  "relevant_devices": ["living_room_light", "kitchen_light"],
  "actions_available": {
    "living_room_light": ["turn_on", "turn_off"],
    "kitchen_light": ["turn_on", "turn_off"]
  }
}
```

The LLM's tool schema will contain only these actions for these devices, even if they support additional capabilities.

## What the Capability System Does Not Do

The capability registry does not:
- **Infer capabilities** — It does not scan device code to discover what it can do
- **Validate semantics** — It does not check if the LLM's choice is the right one (that's the user's responsibility)
- **Prevent hardware failures** — It does not prevent invalid commands due to hardware state
- **Detect hallucinations** — It does not detect if the LLM selected a valid but wrong action

These responsibilities belong to:
- **Hardware state validation** (firmware)
- **Semantic correctness** (user, optionally AI reasoning)
- **Behavioral anomaly detection** ([Planned — Not Implemented](/reference/planned-features))

## Verification and Conformance

A conformance test suite will define minimum behaviors for capability registry compliance. Implementations must:
- Support capability declaration at provisioning
- Enforce capability boundaries at execution
- Log all capability changes
- Support schema validation of action parameters
- Pass conformance test suite to be certified as HxTP-compatible
