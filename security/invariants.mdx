# Architectural Invariants

These are the rules that define the Hestia Labs system. Each invariant must be enforced by architecture, not by convention. If any invariant is violated, the entire security model is compromised.

## I-1: Commands Are Not Executable Until Signed

**Definition:** A command envelope cannot transition from `pending` to `queued` or `executing` without bearing valid signatures.

**Enforcement:** The Dispatch Pipeline (Stage 4: Authority Validation) rejects any command envelope where:
- Planner signature is missing or invalid
- For safety-critical commands, Safety Service signature is missing or invalid

**Why It Matters:** An unsigned command could be locally crafted or replayed. Without this invariant, the AI layer could execute commands directly.

**Verification:** Audit log must contain entry of type `SIGNATURE_VALIDATION` for every command. If a command is executed without this entry, the system has been compromised.

---

## I-2: The Planner Never Has Direct Signing Authority

**Definition:** The HX47 planner process never holds a private key. It cannot sign commands.

**Enforcement:**
- HX47 runs in a distinct process with no access to signing keys
- The Authority Layer (Planner signer) is a separate microservice
- IPC between HX47 and Planner Signer is gated by capability validation
- Capabilities must be declared before any command can be signed

**Why It Matters:** If the LLM process is compromised, it cannot forge signed commands without first compromising the separate Authority Layer. This is defense-in-depth.

**Verification:** Code review of HX47 initialization — no key loading, no cryptographic libraries, no file I/O to `/etc/hestia/keys/`.

---

## I-3: Every Capability Must Be Declared Before Execution

**Definition:** A device cannot execute an action unless that action is listed in its current Capability Manifest.

**Enforcement:**
- During device claim (provisioning), a Capability Manifest is submitted
- The manifest is hash-chained in the audit log
- The Dispatch Pipeline (Stage 6: Capability Routing) rejects any action not in the manifest
- Manifest updates are audited commands, not ambient configuration

**Why It Matters:** This is the core defense against the LLM executing arbitrary operations. The set of allowed actions is finite, declared, and auditable.

**Verification:** 
- Every command in the audit log must map to an action in the device's manifest at the time of execution
- If a command executed an action not in the historical manifest, the system has been compromised

**Related:** T-2 in Trust Boundaries

## I-4: The Safety Service Has Its Own Signing Key

**Definition:** The Safety Service is a cryptographically independent entity. Its signature cannot be forged by compromising any other component.

**Enforcement:**
- Safety signing key is stored in hardware (HSM or secure enclave) separate from other keys
- OPA policy evaluation cannot produce a signature; it can only decide whether to sign
- The Safety Service signs only if the policy returns `true` AND the command structure is valid
- Dual-signature commands require both Planner and Safety signatures

**Why It Matters:** If only the Planner signs, a compromised Planner can forge any command. The Safety Service provides a second opinion from a different trust domain.

**Verification:** 
- Examine cryptographic signature chains — each command for safety-critical actions must have two distinct Ed25519 signatures from different keypairs
- Audit log must show `SAFETY_SERVICE_SIGNATURE` entries distinct from `PLANNER_SIGNATURE` entries

---

## I-5: Audit Log Is Hash-Chained and Append-Only

**Definition:** Every command entry in the audit log is cryptographically linked to the previous entry. No entry can be modified, deleted, or reordered without breaking the chain.

**Enforcement:**
- Each audit log entry contains: `sha256(previous_entry || current_entry_fields)`
- The audit log is stored on immutable storage (append-only MQTT topic, DynamoDB with versioning, or blockchain)
- Reading the log backwards should reveal a continuous chain to entry 0 (genesis block)

**Why It Matters:** An attacker who compromises the system cannot erase traces of the attack. The immutable chain is forensic evidence.

**Verification:**
- Read the entire audit log
- For N entries, verify entry[i].hash == sha256(entry[i-1] || entry[i].fields) for all i > 0
- If any hash validation fails, data has been tampered with

---

## I-6: Dry-Run Commands Do Not Execute Side Effects

**Definition:** A command with `intent.mode = dry_run` produces predictions but makes no state changes.

**Enforcement:**
- The Safety Service evaluates dry-run queries and returns `{"status": "ok", "predictions": {...}, "performed": false}`
- The system never persists state changes from dry-run commands
- Dry-run must never call device actuators, database writers, or external APIs

**Why It Matters:** Users can test commands before committing. The system proves predictability.

**Verification:**
- Query the system with a dry-run command (e.g., "turn on the lights, dry-run")
- Verify that lights did not actually turn on
- Check audit log — entry should show `"mode": "dry_run"` and `"executed": false`

---

## I-7: The Device Manifest Cannot Be Modified Without an Audited Command

**Definition:** The set of capabilities a device advertises is part of the command stream, not ambient configuration.

**Enforcement:**
- Manifest updates are commands issued through HxTP, not configuration file edits
- Each manifest is versioned with a hash
- Manifest updates appear in the audit log with `type: "MANIFEST_UPDATE"`
- Devices cannot update their own manifests (no local file write permission)

**Why It Matters:** If manifests can be edited externally, the capability boundary is meaningless.

**Verification:**
- Attempt to SSH into a Helix Node and edit its manifest file directly
- System should either reject writes or flag the change in audit log as anomalous
- The canonical manifest for execution is always the one recorded in the audit log, not local files

---

## I-8: Certificate Private Keys Are Never Logged

**Definition:** Private keys used for mTLS, signing, or encryption must never appear in any log, error message, or debug output.

**Enforcement:**
- All cryptographic key operations happen within secure boundaries
- Logging functions strip key material before output
- Error messages must never include key material or certificate details
- HSM-backed keys never leave the HSM

**Why It Matters:** If a key is logged, log aggregation systems expose it.

**Verification:**
- Grep entire log stream for private key patterns (Ed25519, RSA headers)
- Should return zero results
- If key material is found, the system has been compromised

---

## I-9: Confirmation Gates Cannot Be Bypassed for Safety-Critical Commands

**Definition:** Commands marked `safety_critical = true` require dual-signature policy gates, and the user cannot force execution without confirmation.

**Enforcement:**
- A safety-critical command enters `pending_confirmation` state
- The Gateway sends a confirmation request to Helix Control app
- Until the user confirms (within timeout), the command waits
- Timeout is enforced — commands do not auto-confirm after expiration
- No override flags, no admin bypass, no emergency mode

**Why It Matters:** Critical commands (e.g., "unlock the front door") should require human validation.

**Verification:**
- Issue a safety-critical command without confirming
- Check command state in audit log — should be `pending_confirmation`
- After timeout (10-60 seconds, depending on configuration), verify command did NOT execute
- Then confirm and verify execution occurs

---

## I-10: Every Commands Must Have a GatewayAuditId

**Definition:** Every command envelope must contain a unique, externally-generated identifier that links execution to user intent.

**Enforcement:**
- The user-facing interface (app, voice, automation) generates a `gatewayAuditId` as a UUID
- This ID is included in the intent envelope
- The Safety Service validates that the ID is present and non-empty
- The audit log records this ID with every entry

**Why It Matters:** This links the originating user action to each step of execution. Without it, commands could be silently originated from elsewhere.

**Verification:**
- Issue a command from Helix Control app, note the command ID
- Query the audit log for that ID
- Should find entries from multiple stages: intent received, validated, signed, executed
- Each entry should have the same `gatewayAuditId`

---

## I-11: Rate Limits Cannot Be Bypassed by Role or Privilege

**Definition:** Every principal (user, automation, external service) is subject to per-device rate limits, regardless of privilege level.

**Enforcement:**
- Rate limiting is applied at the Gateway (before authentication)
- Rate limits apply to all callers: admin, owner, service account
- No role exemption exists
- Hitting a rate limit returns `429 Too Many Requests` with `Retry-After` header

**Why It Matters:** Prevents both accidental and malicious flooding. Even a compromised admin account cannot spam commands.

**Verification:**
- Issue 100 commands in 1 second to a single device
- Verify that after the limit (e.g., 10 per second), subsequent requests return `429`
- Verify this is true even if the client is authenticated as admin

---

## I-12: Failure States Are Always Logged Before Recovery

**Definition:** When the system enters a failure state (LLM crash, Safety Service offline, device unreachable), the failure is recorded in the audit log before any recovery or retry occurs.

**Enforcement:**
- Failure detection triggers an audit log write of type `FAILURE_EVENT`
- The audit log entry includes stack trace, timestamps, affected command ID
- Recovery procedures (retry, timeout, escalation) only proceed after logging completes
- Logs capture the transition from failed state to recovery state

**Why It Matters:** Forensics can reconstruct exactly when, how, and why the system failed.

**Verification:**
- Stop the Safety Service or HX47
- Issue a command that should reach it
- Check audit log — should contain `FAILURE_EVENT` entry
- Verify timestamp of log entry vs. actual failure time are within 1 second

## Summary Table

| Invariant | Enforced By | Verification Method |
|-----------|-------------|-------------------|
| I-1: No unsigned commands | Dispatch Pipeline Stage 4 | Query audit log for SIGNATURE_VALIDATION entries |
| I-2: Planner has no signing key | Architecture (process isolation) | Code review + no keys in HX47 process |
| I-3: Capabilities pre-declared | Dispatch Pipeline Stage 6 capability routing | Map execution actions to historical manifests |
| I-4: Safety Service has own key | HSM + Ed25519 keypair | Two distinct signatures per safety-critical command |
| I-5: Audit log hash-chained | Chain of SHA256 hashes | Verify hash continuity backward to genesis |
| I-6: Dry-run has no side effects | Safety Service output filtering | Run dry-run, verify device state unchanged |
| I-7: Manifest immutable except via commands | Only HxTP updates permitted | Attempt file write, verify rejection or anomaly log |
| I-8: Keys never logged | Logging layer filtering | Grep logs for key patterns, should be zero |
| I-9: Confirmation gates mandatory | Dispatch Pipeline gating | Issue safety-critical command, verify pending state |
| I-10: GatewayAuditId always present | Schema validation | Query audit log, verify ID in every entry |
| I-11: Rate limits universal | Gateway rate limiter | Hit limit as admin, verify 429 response |
| I-12: Failures logged before recovery | Failure handler in Gateway | Stop service, issue command, verify failure log |

## How to Audit These Invariants

For a **security review** of this system:

1. **Code Review:** Search for violations of I-2, I-7, I-8 in source code
2. **Cryptographic Validation:** Verify I-1, I-4, I-5 by examining command envelopes and signatures
3. **Operational Test:** Verify I-3, I-6, I-9, I-11, I-12 by running test commands
4. **Log Analysis:** Verify I-5, I-10, I-12 by examining the audit log structure

If all 12 invariants hold, and the Dispatch Pipeline enforces all validation stages in order, the system is provably secure against arbitrary command execution.
