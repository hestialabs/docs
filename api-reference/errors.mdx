# Error Codes

All Hestia Labs REST API errors follow a standard JSON structure:

```json
{
  "status": "error",
  "error_code": "ERR_CODE_NAME",
  "message": "Human-readable description",
  "details": {
    "field": "optional_field_context"
  },
  "http_status": 400
}
```

---

## Implemented Error Codes

Error codes currently returned by the 6 implemented REST endpoints.

### ERR_INVALID_REQUEST (400)

**Service:** All endpoints  
**Meaning:** Request body is malformed, missing required fields, or invalid JSON syntax

**Recovery:**
1. Verify JSON syntax is valid
2. Check all required fields are present
3. Review endpoint documentation for schema

---

### ERR_UNAUTHORIZED (401)

**Service:** All endpoints  
**Meaning:** Missing or invalid JWT Bearer token

**Recovery:**
1. Include `Authorization: Bearer <token>` header
2. Verify token has not expired
3. Obtain new token from auth service

---

### ERR_FORBIDDEN (403)

**Service:** All endpoints  
**Meaning:** Token is valid but user lacks required role/permission

**Recovery:**
1. Check user's assigned role (admin, operator, viewer)
2. Verify endpoint requires that role
3. Request role upgrade from administrator

---

### ERR_NOT_FOUND (404)

**Service:** `/device/:id/*`  
**Meaning:** Device ID does not exist or was already revoked

**Recovery:**
1. Verify device_id spelling
2. List devices: `GET /devices`
3. Check if device was revoked: `GET /device/:id/key-history`

---

### ERR_DEVICE_NOT_REACHABLE (503)

**Service:** `/device/:id/command`  
**Meaning:** Device is offline or MQTT connection failed

**Recovery:**
1. Check device connectivity (WiFi, network)
2. Verify device is powered on
3. Check device logs for connection errors
4. Retry after 30 seconds (exponential backoff)

---

### ERR_RATE_LIMITED (429)

**Service:** `/device/:id/command` and registration endpoints  
**Meaning:** Rate limit exceeded for this operation

**Response includes:**
```json
{
  "error_code": "ERR_RATE_LIMITED",
  "retry_after_seconds": 60
}
```

**Recovery:**
1. Wait the number of seconds in `retry_after_seconds`
2. Implement exponential backoff for retries
3. For registration bottlenecks, contact administrator

## Planned Error Codes (Not Yet Implemented)

These errors exist in the specification but are not yet returned because their corresponding features are not yet implemented.

### ERR_INVALID_MANIFEST (400 - Planned)

**Service:** Manifest Service (not yet implemented)  
**Meaning:** Manifest schema is invalid or missing required fields  
**Status:** [Planned — Not Implemented]

---

### ERR_SIGNATURE_INVALID (401 - Planned)

**Service:** Intent Service (not yet implemented)  
**Meaning:** Planner or Safety signature verification failed  
**Status:** [Planned — Not Implemented]  
**Note:** Currently, signatures are not evaluated at REST layer; devices verify signatures over MQTT/mTLS

---

### ERR_INVALID_CSR (400 - Planned)

**Service:** Provisioning (not yet implemented)  
**Meaning:** Certificate Signing Request is malformed or expired  
**Status:** [Planned — Not Implemented]

---

### ERR_CONFIRMATION_REQUIRED (403 - Planned)

**Service:** Intent Service (not yet implemented)  
**Meaning:** Intent requires user confirmation before execution  
**Status:** [Planned — Not Implemented]  
**Note:** Confirmation flow exists internally but REST endpoint not yet exposed

---

### ERR_SAFETY_APPROVAL_DENIED (403 - Planned)

**Service:** Safety Service API (not yet implemented)  
**Meaning:** Intent rejected by OPA policy evaluation  
**Status:** [Planned — Not Implemented]  
**Note:** Policy enforcement exists at device level; REST-level policy API not exposed

## Implementation Notes

### Current Error Handling

**What's implemented:**
- Request validation (JSON, required fields, schema)
- Authentication (JWT token check)
- Authorization (role-based access control)
- Device lookup and existence checks
- Device connectivity checks
- Rate limiting and backoff

**What's not yet implemented:**
- Planner Ed25519 signature validation
- Manifest schema validation at REST layer
- OPA policy evaluation at REST layer (happens at device level)
- Provisional CSR validation
- User confirmation state machine at REST layer

### Device-Level Error Handling

Errors that occur at the **device firmware level** (after command dispatch) include:
- Signature validation failures (Planner + Safety)
- Manifest capability checks
- Action parameter validation
- Execution timeouts
- Device-specific errors (hardware failure, sensor offline)

These are returned via the command result, not as HTTP errors.
